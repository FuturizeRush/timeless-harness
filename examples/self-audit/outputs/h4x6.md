# Incident verdict

**Failure — active customer-impacting payment-integrity incident.** The green deployment and unit tests are false-success signals: the test removes the real latency, and the dashboard measures HTTP `200`s rather than charges. The decisive evidence is that one logical request can create two persisted charges and customers have seen duplicates.

**Reject the proposed patch.** Returning `200 {"status":"pending"}` does not prevent either in-flight attempt from charging. It also makes an unresolved outcome look successful, further corrupts the dashboard, and hides the condition operators need to detect. Even if a `200` suppresses a retry in some paths, it cannot safely resolve an already-started or ambiguously completed charge.

## Smallest safe repair

1. **Contain now:** stop the rollout and roll back. Verify the rollback using persisted charges and the provider ledger. If it does not immediately restore one charge per logical request, stop accepting new charges until the invariant is enforced; temporary unavailability is safer than duplicate billing. Disable automatic retries for this endpoint where possible, preserve logs, identify affected transactions, and reverse confirmed duplicate charges with customer notification.
2. **Fix the charge boundary:** require one stable idempotency key per logical charge. Before calling the provider, atomically create or claim a canonical operation under a unique constraint. Forward the same key to a provider that supports idempotency. Every retry must join that operation and return its existing pending or final result—never start another charge.
3. **Handle ambiguity honestly:** after a provider timeout, reconcile the same operation by provider lookup. Return `202 Accepted` with an operation ID only after durable acceptance; never report pending work as `200` success.
4. **Repair detection:** dashboard and alerts must compare logical operations, canonical persisted charges, and provider transactions. HTTP response counts are diagnostic only.

## Evidence required before release

On the exact release build, run an end-to-end fault test through the real gateway and provider sandbox (or equivalent controllable provider), including the observed 10-second timeout, first completion at 12 seconds, retry completion at 8 seconds, concurrent duplicates, lost responses, and process restart.

Each case must prove:

- exactly one provider charge and one canonical persisted charge per idempotency key;
- every attempt resolves to the same operation, with attempts still auditable;
- pending, success, and failure responses retain honest semantics;
- reconciliation completes ambiguous operations without creating another charge;
- persisted/provider mismatches alert, and rollback/recovery works.

Run this complete protocol three times against the same unchanged build, then canary it while monitoring provider and persisted-charge invariants. Any duplicate, orphaned pending operation, misleading success, or changed build resets the release gate.
