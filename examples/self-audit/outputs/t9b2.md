## Incident verdict

**Outcome: failure; active customer-impacting incident.** The release is green only by proxy. The real outcome is duplicate persisted charges, confirmed by customer reports. The dashboard is also giving false reassurance because it counts HTTP `200`s rather than charge side effects.

**Reject the proposed patch.** Returning `200 {"status":"pending"}` does not prevent either attempt from persisting a charge and does not correct retry behavior. It hides the timeout, tells clients and monitoring that the request succeeded, and makes the misleading dashboard look healthier while duplicate-charge risk remains.

## Smallest safe repair

1. **Contain now:** roll back the release. Treat rollback as containment, not proof of a fix. Until the rollback is verified, pause new charge creation or disable the gateway retry for `/charge`; keep status/read paths available. Identify duplicate charges from the incident window, stop further capture where possible, and reconcile/refund affected customers.
2. **Repair at the highest-leverage boundary:** require one stable idempotency key per logical charge and preserve it across gateway retries. Atomically reserve that key before any charge side effect, enforce a database uniqueness constraint, and return the existing result or honest `pending` state for repeats. Pass the same key to the charge provider if it supports idempotency. Concurrent attempts must not both reach the provider or persist separate charges.
3. **Restore honest timeout semantics:** either complete within the gateway deadline or use an explicit asynchronous contract such as `202 Accepted` plus a status endpoint. Align gateway retry rules with that contract. Do not label unknown or pending work as `200` success.
4. Change monitoring to count logical charge outcomes and duplicate-key suppression, with alerts for timeouts, retries, duplicate attempts, and persisted/provider mismatches.

## Evidence required before release

On the exact release candidate, run end-to-end tests through the real gateway behavior with provider delays below and above 10 seconds, including the observed 12-second first attempt/8-second retry and concurrent duplicates. For every case, prove from gateway logs, service logs, provider records, and database rows that one idempotency key produces at most one provider charge and one persisted logical charge, while clients receive an honest final or pending status.

Also prove rollback recovery, retry-key preservation, atomic behavior under races, monitoring accuracy, and a completed reconciliation report for the incident window. Run this complete protocol three times against the same unchanged candidate; any failure or code/config change resets the sequence. Promote only after all three passes are clean.
