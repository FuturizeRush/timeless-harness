## Verdict

This release is a failed, customer-impacting payment incident, despite green deployment checks and unit tests. Gateway timeouts create two live attempts, and both can persist a charge. Reported duplicate charges confirm actual financial harm. The `200` dashboard is not a health signal because it measures responses rather than persisted charges.

Reject the proposed `200 {"status":"pending"}` patch. It neither makes persistence idempotent nor changes retry behavior, so duplicate charges remain possible. It also reports success without a durable pending-state contract or status lookup and makes the existing dashboard look healthier while hiding unresolved payment outcomes.

## Smallest safe repair

1. Contain harm now: stop the affected release from taking new charges. Use the rollback if the prior version is verified not to exhibit this duplicate-charge path; otherwise temporarily disable `/charge` or gateway retries while preserving a clear failure response. Drain in-flight requests before reopening.
2. Identify all logical payments with multiple persisted/provider charges, prevent further capture where possible, refund confirmed duplicates, and notify support/affected customers. Reconcile using payment records and provider transaction IDs—not HTTP status counts.
3. Add durable idempotency at the charge boundary. Require one idempotency key per logical payment and reuse it on every retry. Atomically claim that key with a database uniqueness constraint, persist a state/result, and return the same result (or the same in-progress status) to concurrent and later attempts. Pass the key to the provider if it supports idempotency. This is the highest-leverage fix because it makes retries safe even when latency and timeouts recur.
4. Add metrics and alerts for persisted charges per logical payment, idempotency conflicts/replays, provider outcomes, latency, and unresolved states.

## Evidence required before release

In a production-like environment, exercise provider latency below and above the 10-second gateway timeout, including the observed 12-second first attempt and an 8-second retry. For the same idempotency key, test sequential and concurrent retries plus process restart/crash boundaries. Prove from database rows and provider transaction records that exactly one charge is created, every retry receives a consistent outcome, and no request is falsely reported complete.

Also verify duplicate reconciliation, rollback/disable controls, and new persisted-charge dashboards. Release by canary only after these checks pass; monitor both service and provider records through the retry window before expanding traffic.
