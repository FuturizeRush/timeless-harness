## Verdict: failure

The release is unsafe. Customers are being charged twice. Green deployment/unit tests do not cover the real latency and retry path, and the dashboard measures HTTP responses rather than financial side effects, so both are false-success signals.

Reject the proposed `200 {"status":"pending"}` patch. It does not prevent two attempts from reaching the provider or persisting two charges. It can also turn an unresolved payment into a successful-looking response, further corrupting the dashboard and hiding incidents. “Pending” is valid only when it refers to one durably recorded operation with a reliable way to resolve its final state.

## Contain now

- Start the available rollback. Pause `/charge` traffic or gateway retries until the rollback target is confirmed not to contain this duplicate path and all old-version requests are drained.
- Identify every request in the incident window by customer, amount, provider charge, and retry lineage. Reconcile against the provider ledger—not HTTP `200` counts—then reverse confirmed duplicates and notify affected customers.
- Track remaining in-flight or ambiguous charges until each has one resolved outcome.

## Smallest safe repair

Make idempotency the atomic boundary for a logical charge. Require one stable idempotency key that the gateway reuses on every retry. Before calling the provider, atomically create a charge-intent row protected by a unique constraint on that key. Only the creator may execute the provider call; concurrent or later attempts read and return the existing operation state/result. Pass the same key to the provider when supported. Persist explicit `processing`, `succeeded`, and `failed/unknown` states so a crash or timeout cannot trigger a blind second charge; reconcile `unknown` with the provider before retrying.

Also replace the mocked 100 ms assumption with timeout and retry coverage, and measure unique persisted/provider charges, duplicate attempts blocked, latency, retries, and unresolved operations.

## Release evidence

On the exact candidate build, the complete end-to-end protocol must:

- Force the first attempt past 10 seconds and the retry to overlap/finish sooner; assert one provider charge, one persisted charge, and one stable logical result.
- Cover sequential and concurrent retries plus crashes/timeouts before and after provider acceptance; assert duplicates are blocked and ambiguous states are reconciled without another charge.
- Verify honest response semantics, provider-ledger/database agreement, alerts, dashboards, and rollback.

Run that entire protocol three times against the same unchanged build; every pass must cover every item. Start a monitored canary only if all three passes are clean; any fix restarts the sequence.
