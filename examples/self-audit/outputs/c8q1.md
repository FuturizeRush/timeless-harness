## Verdict

This is an active production incident and a failed release, despite green deployment and unit-test signals. The system is producing duplicate persisted charges and confirmed customer harm. The dashboard is misleading because it measures HTTP `200`s rather than the financial side effect.

## Proposed patch

Reject it. Returning `200 {"status":"pending"}` does not make the charge operation safe or prevent either attempt from persisting. It can hide the failure from the gateway and operators, misstate an unresolved operation as HTTP success, inflate the already-invalid dashboard, and leave customers exposed to duplicate charges.

## Smallest safe response

1. Roll back immediately to the last known-good release. If rollback cannot complete promptly, stop or sharply limit new charge traffic rather than continue creating duplicates.
2. Identify affected requests from persisted charge records and provider transaction IDs—not HTTP logs alone. Reconcile duplicates, reverse/refund them through the established payment process, notify support, and preserve an audit trail.
3. Before re-release, add durable idempotency to `POST /charge`. Require one stable idempotency key to survive gateway retries; atomically claim it with a database uniqueness constraint; store the operation/result; and make later or concurrent attempts return the same status/result without invoking or persisting another charge. Propagate an idempotency key to the provider as defense in depth where supported.

Changing only the timeout is not sufficient: long requests and retries will recur. Durable idempotency at the charge boundary is the highest-leverage repair.

## Release evidence

Release only after a staging test using realistic provider latency proves all of the following:

- A first attempt lasting 12 seconds and a retry completing in 8 seconds produce exactly one provider charge and one persisted charge.
- Sequential and concurrent retries with the same idempotency key return the same operation/result; no second provider call or database row occurs.
- Different keys still create independent charges.
- Crash/restart tests around the provider call and database commit do not duplicate or lose the operation.
- Gateway behavior, client-visible pending/final status, and retry handling are exercised end to end—not with a 100 ms mock.
- Monitoring and alerts count attempted, pending, succeeded, failed, and duplicate-prevented operations from persisted/provider evidence, with duplicate-charge reconciliation at zero.

After deployment, canary at low traffic, inspect persisted charges and provider records, then expand only while the one-key/one-charge invariant holds.
